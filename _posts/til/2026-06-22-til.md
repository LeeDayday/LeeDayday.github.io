## 2026-06-22(월) TIL

### Struct와 Instance

- `struct`도 `class`처럼 `()`를 사용해서 값을 생성할 수 있다.

```swift
struct MyStack {
    var data: [Int] = []
}

let stack = MyStack()
```

- 정확한 표현은 `객체를 만들었다`보다 `인스턴스를 만들었다`가 더 좋다.
- Swift에서 `struct`는 value type이고, `class`는 reference type이다.
- `struct` 내부의 저장 프로퍼티를 메서드에서 변경하려면 `mutating`이 필요하다.

```swift
mutating func insertValue(input: Int) {
    data.append(input)
}
```

## Generic

- Generic은 타입 안정성을 유지하면서 여러 타입에 대응하는 방법이다.
- Python처럼 타입이 느슨해지는 것이 아니라, 사용하는 시점에 타입이 확정된다.
- 정의하는 곳에서는 타입 파라미터를 선언한다.

```swift
struct MyStack<T> {
    var data: [T] = []
}
```

- 사용하는 곳에서는 실제 타입을 넣는다.

```swift
let stringStack = MyStack<String>()
let intStack = MyStack<Int>()
```

- `T`는 값 인자가 아니라 타입 파라미터다.
- 한 번 `MyStack<String>`으로 정해지면 `String`만 담을 수 있다.

### SwiftUI Layout

- SwiftUI layout은 부모가 자식의 크기를 강제로 정하는 방식이 아니다.
- 부모가 자식에게 가능한 크기를 제안하고, 자식이 필요한 크기를 다시 보고한다.

```text
부모 -> 자식: 이 정도 공간 안에서 그려볼래?
자식 -> 부모: 나는 이만큼 필요해.
부모: 그럼 여기 배치할게.
```

- 크기 제안은 위에서 아래로 내려간다.
- 실제 필요한 크기는 아래에서 위로 올라간다.

예시:

```swift
Text("Hello world")
    .padding(20)
    .background(Color.red)
```

- `padding(20)`은 자식인 `Text`에게 줄 수 있는 공간을 padding만큼 줄여서 제안한다.
- `Text`가 자기 크기를 정하면, `padding`은 그 크기에 상하좌우 20을 더해 자신의 크기로 보고한다.
- `background`는 앞에서 결정된 크기만큼 배경을 칠한다.
- modifier 순서에 따라 감싸는 구조와 레이아웃 결과가 달라진다.

### SwiftUI State Management

- SwiftUI에서 중요한 것은 무조건 MVVM을 적용하는 것이 아니라, 상태의 소유권과 공유 범위를 잘 정하는 것이다.
- 상태가 필요한 가장 좁은 범위에 두는 것이 좋다.

```text
@State
내 View가 소유하는 단순 상태

@Binding
부모의 상태를 자식이 받아서 수정

@StateObject
View가 ObservableObject를 생성하고 소유

@ObservedObject
외부에서 받은 ObservableObject를 관찰

@EnvironmentObject
상위 환경에 주입된 공유 객체를 사용
```

- `ObservableObject`는 protocol이다.
- `@ObservedObject`는 property wrapper다.
- `@EnvironmentObject`는 편하지만 의존성이 숨겨지기 쉽다.
- 테스트나 Preview에서 필요한 환경 객체를 매번 주입해야 하므로 오버헤드가 생길 수 있다.

```swift
ProfileView()
    .environmentObject(SessionStore())
```

- 정말 전역적으로 공유되어야 하는 상태에만 `@EnvironmentObject`를 쓰는 편이 좋다.

### 알고리즘 Talk #1 - DFS와 DP의 연결고리

### 계단 오르기 문제

- 현재 위치는 `0`.
- 한 번에 `+1칸` 또는 `+2칸` 이동할 수 있다.
- `N`칸에 도착하는 방법의 수를 구하는 문제.

예시:

```text
N = 3

1 + 1 + 1
1 + 2
2 + 1

정답 = 3
```

### 점화식

- `n`번째 칸에 도착하는 방법은 두 가지로 나눌 수 있다.
- `n - 1`칸에서 `1칸` 이동
- `n - 2`칸에서 `2칸` 이동

```swift
answer[idx] = answer[idx - 1] + answer[idx - 2]
```

초기값:

```swift
answer[0] = 1
answer[1] = 1
```

- `answer[0] = 1`은 "아무것도 하지 않고 0칸에 있는 방법 1가지"를 의미한다.

### DFS 관점

- 같은 문제를 DFS로도 표현할 수 있다.

```swift
dfs(position + 1)
dfs(position + 2)
```

- DFS는 현재 상태에서 가능한 선택지를 따라 내려간다.
- 계단 문제에서 상태는 `현재 몇 번째 칸인가`이다.
- 행동은 `1칸 이동` 또는 `2칸 이동`이다.

### DFS에서 DP로 이어지는 흐름

- DP는 갑자기 등장하는 것이 아니다.
- DFS로 풀다 보면 같은 상태를 여러 번 계산한다.
- 그 중복 계산을 저장해서 줄이면 memoization이 된다.
- memoization을 체계화하면 DP로 이어진다.

```text
DFS
-> 중복 계산 발견
-> Memoization
-> DP
```

- `0 -> 2`와 `0 -> 1 -> 2`는 둘 다 결국 `dfs(2)`를 호출한다.
- 즉, `2에서 시작했을 때의 결과`를 여러 번 계산하고 있다.
- 이 값을 `memo[2]`에 저장하면 중복 계산을 줄일 수 있다.

### DFS는 그래프 전용이 아니다

- DFS를 `그래프 탐색 알고리즘`으로만 외우면 범위가 좁아진다.
- 더 정확히는 `상태 공간 탐색`에 가깝다.

예시:

- 계단 문제: 현재 칸에서 `1칸 이동` / `2칸 이동`
- 조합 문제: 현재 원소를 `선택` / `선택 안 함`
- 그래프 문제: 현재 정점에서 `인접 정점 방문`

### DFS의 인자

- DFS의 인자는 보통 `현재 상태`를 의미한다.

```swift
dfs(position) // 현재 몇 번째 칸인가
dfs(index)    // 현재 몇 번째 원소를 보고 있는가
dfs(node)     // 현재 어떤 정점에 있는가
```

- DFS를 볼 때는 `이 인자가 현재 상태를 제대로 표현하고 있는가?`를 먼저 생각한다.

### 전역 변수 vs 반환값

- 모든 경우를 순회하면서 개수를 세는 문제는 전역 변수를 쓰기 쉽다.

```swift
var answer = 0
answer += 1
```

- 예시: 순열, 조합, N-Queen, 백트래킹
- 특정 상태의 결과를 계산하는 문제는 반환값을 쓰는 편이 자연스럽다.

```swift
func dfs(...) -> Int
```

- 반환값 기반 DFS는 memoization과 연결하기 쉽다.

### DFS는 Top-Down인가 Bottom-Up인가

- 탐색은 Top-Down이다.
- 계산은 Bottom-Up처럼 올라오며 완성된다.
- `dfs(0)`에서 시작하므로 탐색은 `0 -> 1 -> 2 -> 3`처럼 내려간다.
- 반환값은 `3 -> 2 -> 1 -> 0`처럼 아래 상태에서 위 상태로 올라온다.

### 오늘 배운 것

- `struct`도 `()`로 인스턴스를 생성할 수 있다.
- Generic은 타입을 느슨하게 만드는 기능이 아니라, 타입 안정성을 유지하면서 재사용성을 높이는 기능이다.
- SwiftUI layout은 위에서 아래로 크기를 제안하고, 아래에서 위로 필요한 크기를 보고하는 방식이다.
- SwiftUI에서는 MVVM 자체보다 적절한 상태 관리 구조가 더 중요하다.
- DFS는 그래프 전용 알고리즘이 아니라 상태 공간 탐색 방식이다.
- DP는 DFS 과정에서 발견한 중복 계산을 제거하는 최적화 기법으로 볼 수 있다.
